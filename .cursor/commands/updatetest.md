# 测试基线的更新

用于在新增功能或行为变更后，依据代码改动评估并更新基线测试用例，确保回归覆盖始终与当前能力一致。

## 目标

- 保证“代码能力”与“测试基线”同步。
- 避免新增功能无测试、旧测试与现行为不一致。
- 每次迭代均可快速判断是否需要补测。

## 触发时机（满足任一即触发）

- 新增/删除/修改 API 路由、请求参数、响应字段。
- 新增工具、Skill、过滤器、MCP 服务器或配置开关。
- 调整流式与非流式行为（如 chunk 类型、错误处理、会话要求）。
- 超时、重试、并发、熔断等稳定性策略变更。
- 错误码、错误文案、降级逻辑发生变化。

## 更新流程

### 1) 先扫代码，确认能力变化

推荐在项目根目录执行（可按需增减关键词）：

```bash
cd /mnt/d/workspace/prodject/zero-agent
rg -n "@app\\.(get|post)|/agents|/health|/tools|stream|StreamChunk|function_call|message_history|session_id|skills|mcp|timeout|concurrency" api orchestration config filters tools skills
```

输出解读规则：
- 命中 API 声明：检查是否新增/变更接口及参数。
- 命中 `stream` / `StreamChunk`：检查流式事件结构是否变化。
- 命中 `session_id`：检查必填约束与错误分支是否变化。
- 命中 `skills` / `mcp`：检查能力加载、降级、可见性是否变化。

### 2) 改动映射到测试项（必须落地到用例）

按“改动类型 -> 必测项”补全测试：

- API 结构变更  
  - 成功用例 + 至少 1 个错误分支（缺参/非法值/不存在资源）。
- 流式链路变更  
  - 校验 SSE 连续输出、chunk `type`、done 终止语义。
- 非流式链路变更  
  - 校验 OpenAI 兼容结构与 `finish_reason`。
- Skill 相关变更  
  - 校验 Skill 上下文注入是否生效（流式与非流式都要覆盖）。
- MCP/外部依赖变更  
  - 校验失败可降级且不阻断服务启动。
- 并发/超时/重试变更  
  - 增加轻量并发与超时边界验证。

### 3) 更新基线文档

优先更新：
- [testagent.md](mdc:.cursor/commands/testagent.md)

更新内容至少包含：
- 新增或变更能力说明。
- 对应 curl 用例（成功 + 错误分支）。
- 通过标准 Checklist。
- 当前已知行为（如“仅流式生效”变为“流式/非流式一致”）。

### 4) 执行并记录结果

- 按文档顺序跑冒烟 -> 重点链路 -> 错误分支 -> 并发稳定性。
- 对每个新增/变更用例记录“请求、响应、结论”。
- 若行为与预期不一致，先修文档或修代码，避免基线失真。

## 判定规则（是否必须更新基线）

满足以下任一，判定为“必须更新”：
- 新增能力可被用户感知（接口、字段、输出结构、错误码）。
- 旧用例无法验证新逻辑或关键分支。
- 现有文档中的“已知行为”与当前代码不一致。

若仅为内部重构且外部行为完全不变：
- 可不新增用例；
- 但建议补一条“等价重构”说明，避免后续误判。

## 最小更新模板（可直接复制）

```md
### 变更项
- [ ] 变更名称：
- [ ] 影响范围：API / Stream / Non-Stream / Skill / MCP / Timeout / Concurrency

### 用例补充
- [ ] 成功用例：
- [ ] 错误分支：
- [ ] 边界场景：

### 断言
- [ ] 状态码正确
- [ ] 响应结构字段正确
- [ ] 关键语义正确（如 session_id 必需、chunk type、finish_reason）

### 结论
- [ ] 已更新基线文档
- [ ] 已完成回归验证
```

## 常见遗漏

- 只更新流式测试，漏掉非流式回归。
- 新增必填字段后，未补“缺字段错误分支”。
- 只验证功能成功，不验证降级路径（MCP 下线/超时）。
- 代码已变更，但 Checklist 仍停留在旧描述。

## 输出要求（提交前自检）

- 文档中每个新增能力都能找到对应测试项。
- 每个关键接口至少有 1 条失败分支测试。
- Skill、Tool、MCP 三类能力均有可执行验证路径。
- 基线文档可由他人直接执行，不依赖口头说明。
